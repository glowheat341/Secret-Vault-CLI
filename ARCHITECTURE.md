## 🎯 Общая архитектура

Secret Vault CLI построен на трёхслойной архитектуре:

```
┌─────────────────────────────────────────────────────────┐
│                    CLI Layer (cli.ts)                   │
│  • Commander.js - парсинг команд                        │
│  • Inquirer - интерактивные промпты                     │
│  • Chalk - цветной вывод                                │
│  • Clipboardy - работа с буфером обмена                 │
│  • Ora - спиннеры и индикаторы прогресса                │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                 Business Logic (vault.ts)               │
│  • Управление секретами (CRUD операции)                 │
│  • Поиск и фильтрация                                   │
│  • Управление категориями                               │
│  • Импорт/экспорт                                       │
│  • Управление состоянием (lock/unlock)                  │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│              Crypto Layer (encryption.ts)               │
│  • AES-256-GCM шифрование                               │
│  • Scrypt деривация ключей                              │
│  • Генерация случайных значений                         │
│  • Аутентификация данных                                │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│              Storage Layer (filesystem)                 │
│  • ~/.secret-vault/vault.enc                            │
│  • Зашифрованный файл в base64                          │
└─────────────────────────────────────────────────────────┘
```

## 🔐 Криптография

### Алгоритм шифрования: AES-256-GCM

**AES-256-GCM** (Advanced Encryption Standard with Galois/Counter Mode) - это:

- **AES-256**: Симметричное блочное шифрование с ключом 256 бит
- **GCM**: Режим работы, обеспечивающий:
  - **Конфиденциальность** (шифрование данных)
  - **Аутентификацию** (проверка целостности)
  - **Защиту от подделки** (authentication tag)

### Константы шифрования

```typescript
const ALGORITHM = 'aes-256-gcm';      // Алгоритм шифрования
const SALT_LENGTH = 32;                // 32 байта (256 бит) - соль для scrypt
const IV_LENGTH = 16;                  // 16 байт (128 бит) - вектор инициализации
const AUTH_TAG_LENGTH = 16;            // 16 байт (128 бит) - тег аутентификации
const KEY_LENGTH = 32;                 // 32 байта (256 бит) - длина ключа
```

### Процесс шифрования (encrypt)

```
┌──────────────────────────────────────────────────────────────┐
│ 1. ГЕНЕРАЦИЯ СЛУЧАЙНЫХ ЗНАЧЕНИЙ                              │
└──────────────────────────────────────────────────────────────┘
   ↓
   • salt = randomBytes(32)    ← Уникальная соль для каждого шифрования
   • iv = randomBytes(16)      ← Уникальный вектор инициализации

┌──────────────────────────────────────────────────────────────┐
│ 2. ДЕРИВАЦИЯ КЛЮЧА (scrypt)                                  │
└──────────────────────────────────────────────────────────────┘
   ↓
   key = scryptSync(password, salt, 32)
   
   Scrypt - это функция деривации ключа (KDF), которая:
   • Преобразует пароль в криптографический ключ
   • Использует много памяти (защита от ASIC/GPU атак)
   • Медленная по дизайну (защита от брутфорса)
   • Использует соль (защита от rainbow tables)

┌──────────────────────────────────────────────────────────────┐
│ 3. ШИФРОВАНИЕ ДАННЫХ (AES-256-GCM)                           │
└──────────────────────────────────────────────────────────────┘
   ↓
   cipher = createCipheriv('aes-256-gcm', key, iv)
   encrypted = cipher.update(data, 'utf8', 'hex')
   encrypted += cipher.final('hex')
   authTag = cipher.getAuthTag()  ← Тег для проверки целостности

┌──────────────────────────────────────────────────────────────┐
│ 4. ФОРМИРОВАНИЕ ИТОГОВОГО БЛОБА                              │
└──────────────────────────────────────────────────────────────┘
   ↓
   result = [salt | iv | authTag | encrypted]
            └─32─┘└16┘└──16──┘└─variable─┘
   
   Все компоненты объединяются в один буфер:
   • salt (32 байта) - для деривации ключа при расшифровке
   • iv (16 байт) - для инициализации дешифратора
   • authTag (16 байт) - для проверки целостности
   • encrypted (переменная длина) - зашифрованные данные

┌──────────────────────────────────────────────────────────────┐
│ 5. КОДИРОВАНИЕ В BASE64                                      │
└──────────────────────────────────────────────────────────────┘
   ↓
   return result.toString('base64')
   
   Base64 для безопасного хранения в текстовом файле
```

### Процесс расшифровки (decrypt)

```
┌──────────────────────────────────────────────────────────────┐
│ 1. ДЕКОДИРОВАНИЕ ИЗ BASE64                                   │
└──────────────────────────────────────────────────────────────┘
   ↓
   buffer = Buffer.from(encryptedData, 'base64')

┌──────────────────────────────────────────────────────────────┐
│ 2. ИЗВЛЕЧЕНИЕ КОМПОНЕНТОВ                                    │
└──────────────────────────────────────────────────────────────┘
   ↓
   salt = buffer[0:32]           ← Первые 32 байта
   iv = buffer[32:48]            ← Следующие 16 байт
   authTag = buffer[48:64]       ← Следующие 16 байт
   encrypted = buffer[64:end]    ← Остальное - зашифрованные данные

┌──────────────────────────────────────────────────────────────┐
│ 3. ДЕРИВАЦИЯ КЛЮЧА (с той же солью)                          │
└──────────────────────────────────────────────────────────────┘
   ↓
   key = scryptSync(password, salt, 32)
   
   Используется та же соль, что была при шифровании

┌──────────────────────────────────────────────────────────────┐
│ 4. РАСШИФРОВКА И ПРОВЕРКА ЦЕЛОСТНОСТИ                        │
└──────────────────────────────────────────────────────────────┘
   ↓
   decipher = createDecipheriv('aes-256-gcm', key, iv)
   decipher.setAuthTag(authTag)  ← Устанавливаем тег для проверки
   
   decrypted = decipher.update(encrypted, 'hex', 'utf8')
   decrypted += decipher.final('utf8')
   
   Если authTag не совпадает или пароль неверный:
   → Выбрасывается исключение
   → Данные не расшифровываются

┌──────────────────────────────────────────────────────────────┐
│ 5. ВОЗВРАТ РАСШИФРОВАННЫХ ДАННЫХ                             │
└──────────────────────────────────────────────────────────────┘
   ↓
   return decrypted  ← JSON строка с секретами
```

```typescript
{
  "version": "1.0.0",           // Версия формата хранилища
  "secrets": [                  // Массив секретов
    {
      "name": "github-token",   // Уникальное имя секрета
      "value": "ghp_xxx...",    // Значение секрета
      "category": "api-keys",   // Категория (опционально)
      "createdAt": "2026-05-18T17:00:00.000Z",  // Дата создания
      "updatedAt": "2026-05-18T17:30:00.000Z",  // Дата обновления
      "metadata": {             // Дополнительные метаданные (опционально)
        "description": "GitHub personal access token",
        "expires": "2027-05-18"
      }
    }
  ]
}
```

### Формат зашифрованного файла

```
~/.secret-vault/vault.enc

После декодирования (бинарный формат):
[32 байта salt][16 байт IV][16 байт authTag][зашифрованный JSON]
```

## 🔒 Безопасность

### Многоуровневая защита

```
┌─────────────────────────────────────────────────────────────┐
│ УРОВЕНЬ 1: Шифрование данных в покое (at rest)              │
├─────────────────────────────────────────────────────────────┤
│ • AES-256-GCM - военный стандарт шифрования                 │
│ • Все данные зашифрованы на диске                           │
│ • Без мастер-пароля данные бесполезны                       │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ УРОВЕНЬ 2: Защита от брутфорса                              │
├─────────────────────────────────────────────────────────────┤
│ • Scrypt - memory-hard функция                              │
│ • Требует много RAM для вычисления                          │
│ • Медленная по дизайну (~100ms на деривацию)                │
│ • Защита от GPU/ASIC атак                                   │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ УРОВЕНЬ 3: Защита от rainbow tables                         │
├─────────────────────────────────────────────────────────────┤
│ • Уникальная случайная соль (32 байта)                      │
│ • Новая соль для каждого шифрования                         │
│ • Невозможно предвычислить хеши                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ УРОВЕНЬ 4: Защита целостности данных                        │
├─────────────────────────────────────────────────────────────┤
│ • GCM authentication tag (16 байт)                          │
│ • Обнаружение любых изменений                               │
│ • Защита от подделки данных                                 │
│ • Автоматическая проверка при расшифровке                   │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ УРОВЕНЬ 5: Защита от повторного использования              │
├─────────────────────────────────────────────────────────────┤
│ • Уникальный IV (16 байт) для каждого шифрования            │
│ • Защита от replay атак                                     │
│ • Невозможность определить одинаковые данные                │
└─────────────────────────────────────────────────────────────┘
```

### Что НЕ защищается

```
❌ Мастер-пароль в памяти
   • Хранится в RAM пока хранилище разблокировано
   • Уязвимо к memory dumps
   • Решение: vault lock после использования

❌ Расшифрованные секреты в памяти
   • Секреты расшифровываются для использования
   • Находятся в RAM во время работы
   • Решение: минимизировать время разблокировки

❌ Буфер обмена
   • Секреты копируются в системный clipboard
   • Могут быть прочитаны другими приложениями
   • Решение: очистить буфер после использования

❌ Keyloggers и malware
   • Могут перехватить ввод мастер-пароля
   • Могут прочитать память процесса
   • Решение: чистая система, антивирус

❌ Физический доступ к разблокированному хранилищу
   • Если vault.lock() не вызван
   • Решение: автоматическая блокировка по таймауту
```

### Threat Model (Модель угроз)

#### ✅ Защищает от:

1. **Кража файла хранилища**
   - Без мастер-пароля данные бесполезны
   - AES-256 практически невзламываем

2. **Брутфорс пароля**
   - Scrypt делает перебор очень медленным
   - 1 попытка ≈ 100ms → 10 попыток/сек
   - Для 8-символьного пароля: годы вычислений

3. **Rainbow table атаки**
   - Уникальная соль для каждого шифрования
   - Невозможно использовать предвычисленные таблицы

4. **Подделка данных**
   - GCM authentication tag обнаружит любые изменения
   - Невозможно модифицировать зашифрованные данные

5. **Утечка одного секрета**
   - Каждый секрет не связан с другими
   - Компрометация одного не влияет на остальные

#### ❌ НЕ защищает от:

1. **Слабый мастер-пароль**
   - "password123" взломается быстро
   - Требование: минимум 8 символов (лучше 16+)

2. **Keylogger на системе**
   - Перехватит ввод мастер-пароля
   - Решение: чистая система

3. **Memory dump атаки**
   - Пароль и секреты в RAM при разблокировке
   - Решение: vault lock после использования

4. **Физический доступ к запущенной системе**
   - Если хранилище разблокировано
   - Решение: блокировка экрана, vault lock

5. **Социальная инженерия**
   - Выманивание мастер-пароля у пользователя
   - Решение: обучение пользователей

## 🎯 Ключевые особенности реализации

### 1. Stateless архитектура

```typescript
// Мастер-пароль хранится только в памяти
private masterPassword: string | null = null;

// При каждой операции:
// 1. Проверяется разблокировка
// 2. Читается файл
// 3. Расшифровывается
// 4. Выполняется операция
// 5. Зашифровывается
// 6. Записывается обратно
```

### 2. Fail-safe операции

```typescript
// Все операции шифрования в try-catch
try {
  const decrypted = this.encryption.decrypt(encrypted, password);
  return JSON.parse(decrypted);
} catch {
  return false;  // Неверный пароль или повреждённые данные
}
```

### 3. Атомарные операции

```typescript
// Чтение → Модификация → Запись в одной транзакции
const data = this.getVaultData();  // Читаем
data.secrets.push(newSecret);      // Модифицируем
this.saveVault(data);              // Сохраняем
```

### 4. Независимость шифрования

```typescript
// Каждое шифрование независимо
encrypt(data, password) {
  const salt = randomBytes(32);    // Новая соль
  const iv = randomBytes(16);      // Новый IV
  // ... шифрование
}

// Экспорт использует отдельный пароль
exportVault(exportPassword) {
  const data = this.getVaultData();  // Расшифровать основным паролем
  return this.encryption.encrypt(    // Зашифровать экспортным паролем
    JSON.stringify(data), 
    exportPassword
  );
}
```

## 🔧 Технические детали

### Используемые библиотеки

```typescript
// Node.js встроенный crypto модуль
import { 
  createCipheriv,      // Создание шифратора
  createDecipheriv,    // Создание дешифратора
  randomBytes,         // Криптографически стойкий RNG
  scryptSync           // Синхронная деривация ключа
} from 'crypto';
```

### Почему именно эти алгоритмы?

**AES-256-GCM:**
- ✅ Стандарт NIST (National Institute of Standards and Technology)
- ✅ Используется в TLS 1.3, IPsec, SSH
- ✅ Аппаратное ускорение (AES-NI на современных CPU)
- ✅ Authenticated encryption (шифрование + аутентификация)
- ✅ Параллелизуемый (быстрый)

**Scrypt:**
- ✅ Специально разработан для защиты паролей
- ✅ Memory-hard (требует много RAM)
- ✅ Защита от GPU/ASIC атак
- ✅ Используется в Bitcoin, Litecoin
- ✅ Рекомендован OWASP

**Альтернативы и почему НЕ они:**

```
❌ AES-CBC - нет встроенной аутентификации
❌ AES-CTR - нет встроенной аутентификации
❌ ChaCha20 - нет аппаратного ускорения на всех платформах
❌ PBKDF2 - слабее против GPU атак чем scrypt
❌ bcrypt - ограничение на длину пароля (72 байта)
❌ Argon2 - не встроен в Node.js crypto
```

## 🎓 Заключение

Secret Vault CLI использует современные криптографические стандарты для обеспечения максимальной безопасности локального хранения секретов. Архитектура построена на принципах:

- **Defense in depth** - многоуровневая защита
- **Fail-safe** - безопасное поведение при ошибках
- **Simplicity** - простота снижает вероятность ошибок
- **Standards compliance** - использование проверенных стандартов
- **Transparency** - открытый исходный код для аудита

Проект подходит для хранения:
- ✅ API ключей и токенов
- ✅ Паролей от сервисов
- ✅ SSH ключей и сертификатов
- ✅ Database credentials
- ✅ Конфигурационных секретов

Не подходит для:
- ❌ Высококлассифицированных государственных секретов
- ❌ Финансовых данных (используйте HSM)
- ❌ Медицинских данных (требуется HIPAA compliance)
- ❌ Shared secrets в команде (используйте HashiCorp Vault)
